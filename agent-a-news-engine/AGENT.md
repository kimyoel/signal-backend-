# AGENT.md — AI 에이전트 규칙서

---

## 🔁 작업 사이클 (매 작업마다 반드시 이 순서로)

### 1단계 — 작업 시작 전: 4개 파일 읽기
새 세션을 열거나 새 작업을 시작하기 전, 아래 4개 파일을 반드시 먼저 읽는다.

| 파일 | 읽는 이유 |
|------|----------|
| `AGENT.md` | 이 프로젝트의 규칙 파악 |
| `plan.md` | 전체 구조와 설계 방향 파악 |
| `context.md` | 기술 선택 근거 및 결정 사항 파악 |
| `tasks.md` | 어디까지 완료됐는지, 다음 작업이 뭔지 파악 |

4개 파일을 읽지 않고 바로 코딩을 시작하지 않는다.

---

### 2단계 — 계획서 제출 (코딩 전)
작업을 시작하기 전, 코드를 한 줄도 작성하지 않고 **자연어로 된 계획서를 먼저 제출**한다.

계획서에는 아래 항목이 포함되어야 한다:
- 이번 작업의 범위 (무엇을 만드는지)
- 단계별 구현 순서
- 새로 생성하거나 수정하는 파일 목록
- 기존 파일을 삭제하거나 교체하는 경우, 그 이유와 백업 여부
- 완료 기준 (어떤 상태가 되면 이 작업이 끝난 건지)

---

### 3단계 — 승인 대기
계획서를 제출한 뒤, **사용자의 명시적인 승인을 받기 전까지 코딩을 시작하지 않는다.**

사용자가 "승인" 또는 "시작해"라고 하면 계획서대로 구현한다.
수정 요청이 오면 계획서를 수정하고 다시 승인을 기다린다.

---

### 4단계 — 계획서대로 구현
승인된 계획서의 범위 안에서만 작업한다.
승인받지 않은 파일을 수정하거나, 범위를 벗어난 작업을 추가하지 않는다.
작업 중 예상치 못한 변경이 필요하면 멈추고 사용자에게 먼저 알린다.

---

### 5단계 — 작업 완료 후: 파일 업데이트
작업이 완료되면 아래 파일들을 업데이트한다.

- `tasks.md` : 완료된 항목을 `[ ]` → `[x]` 로 변경, 완료 날짜 기입
- `context.md` : 작업 중 새로운 기술 결정이 있었다면 그 내용과 이유 추가
- `plan.md` : 설계 방향이 바뀐 경우 업데이트

---

## 🚫 절대 하지 말아야 할 것들

- 계획서 없이 바로 코딩 시작하는 것
- 승인 전에 코드를 작성하는 것
- 승인된 범위를 벗어나 다른 파일까지 수정하는 것
- 기존 파일을 삭제하거나 전체 교체하기 전에 사용자에게 확인하지 않는 것
- API 키, 비밀번호, JWT Secret 등을 코드에 직접 작성하는 것 (반드시 환경변수 사용)
- 작업 완료 후 tasks.md 업데이트를 빠뜨리는 것

## 💬 계획서 제출 형식

계획서는 아래 형식으로 제출한다.

```
## 구현 계획서: [기능명]

**범위**: 이번 작업에서 만들거나 수정하는 것
**단계**:
  1. 
  2. 
  3. 
**생성/수정 파일**: 
**삭제 또는 교체 파일**: (없으면 "없음")
**완료 기준**: 이 상태가 되면 작업 완료로 간주

승인하시면 구현 시작하겠습니다.
```

## 📝 기타 규칙

- 한 번 지시에 한 기능 단위로만 작업한다. 여러 기능을 한꺼번에 진행하지 않는다.
- 에러가 발생하면 추측으로 수정하지 않고, 에러 메시지 전체를 사용자에게 보여주고 확인한다.
- 코드를 이해하기 어려운 부분이 있으면 주석을 한국어로 달아 설명한다.
- 기능 구현과 테스트 코드 작성은 항상 한 세트로 진행한다.
- 각 기능 완성 시 git commit을 한다. (`feat: 기능명` 형식)
